Gui Graph Editor

Gui Graphs describe the basic operaional view on the system under test. The model is based on Petri nets plus several extensions. A Petri net consists of three fundamental elements:

  • Places consitute a partitial state. A place can be filled with multiple tokens. An assignment of tokens the a subset of places of a Petri net is a marking and, thus, a state.
  • Transitions define behavior.
  • (Directed) Arcs can be connect a Place with a Transition but never two entities of the same type. An arc has a weight to define how many tokens are transferred on this specific arc.

We say, transitions get activated if all places from incoming arcs are filled with enough tokens for each arc's weight. Then the transition may fire by removing these tokens and putting new tokens in places that are connected by output arcs in the respective number of the arcs' weight.

Petri nets are not Turing-complete. Therefore, Inhibitor arcs can be used that are special arcs from places to transitions. A transition with with an incoming inhibitor arc can only be activated if the respective place is not marked by a token.

Besides inhibitor arcs, we added some additional extensions for Gui Graphs. Places are provided in two different visual representations and transitions are extended with labels, application conditions and time intervals. We call the special type of our transitions Condition Action Transitions.

Two Types of Places

There are two semantically equal types of places:

  1. Non-widget Nodes and
  2. Widget Nodes that can additionally be associated to screenshots. This can be used to provide an additional orientation for the test engineer.

Using the property view, an additional number of tokens can be set to each place.

Labels

Lables descibe observable behavior that is executed when a transition "fires". Gui Graph labels are defined by the Action Language. Define labels by double clicking on the middle compartment of a condition action transition.

Application Conditions (only in the Adaptivity Testing Version)

Application conditions are taken from so-called Feature Petri nets. They restrict the activation of each transition to a propositional constraint defined in the Rules Language. Define application conditions by double clicking on the upper compartment of a condition action transition.

Time Intervals (only in the Adaptivity Testing Version)

Time intervals are taken from so-called Time Petri nets. They restrict the firering of transitions to a (real) time inside the respective interval. The time counter starts as soon as the transitions becomes enabled (equal to activation in standard Petri nets). You can define a temporal indeterminism with the help of these intervals. They can be changed in the property view after selecting the respecitve transition.

Clocking Transitions

Clocking transitions work like standard transitions of Inhibitor Arc Petri nets plus that they produce logic time in order to control timed environment models (so-called consumers). Clocking transitions have activation time intervals as well. In the property view the amount of produced logical time can be defined as well as a specific consumer. If no consumer is defined, all consumers in the model's resource set are run.
